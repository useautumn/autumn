# CURSOR.md — Cursor Cloud agent instructions

Environment/setup notes for Cursor agents running this repo in a Cursor Cloud VM.
General repo rules live in `AGENTS.md` (generated by `bun sync` from the `ai/`
submodule — see below); this file is the committed, durable doc for the Cursor
Cloud environment. `AGENTS.md`, `.cursor/`, `.claude/`, `.opencode/` are all
git-ignored because they are regenerated by `bun sync`.

## One command to bring up everything: `scripts/setup/cloud-dev.sh`

```sh
bash scripts/setup/cloud-dev.sh
```

This is the single entrypoint that spins up **all required services + the app**.
It is idempotent and does, in order:

1. **dev-services Docker container** — `docker/dev-services.compose.yml` brings up
   Postgres 18 (`:5432`, db `autumn`, `postgres`/`postgres`), Redis Stack (`:6379`)
   and Dragonfly (`:6380`). Only these three services are started (the `ngrok`
   profile is skipped — it needs an authtoken/Infisical). Starts `dockerd` first
   if it isn't already running.
2. **goaws** (`:9324`) — the local SQS server (FIFO queues `autumn.fifo` +
   `autumn-track.fifo`), used in place of ElasticMQ. Binary is fetched once from
   the `admiralpiett/goaws` image via `crane` into `~/.autumn-agent/bin`.
3. **`bun dw setup`** (`scripts/dw/index.ts setup`) — runs `bun install`, inits the
   **`ai` submodule**, checks out its `main` branch, installs its deps and runs
   **`bun sync`** (the "ai sync" that writes `AGENTS.md` + the `.cursor`/`.claude`
   skills/rules/agents). Invoked directly to bypass the Infisical wrapper baked
   into the `bun dw` npm alias.
4. **`server/.env`** — written (if missing) with local-service URLs + generated
   `BETTER_AUTH_SECRET`/`ENCRYPTION_*` secrets.
5. **Migrations** — `bun db migrate --bootstrap` + the balance SQL functions.
6. **`bun dw`** — launches the full dev stack in the foreground.

Provision without launching the app: `bash scripts/setup/cloud-dev.sh --services`,
then run the stack yourself with `bun scripts/dw/index.ts`.

> Do **not** use the older `agent-bootstrap.sh` / `dev:agent` / capy flows here —
> they install local apt Postgres/Redis/ClickHouse + ElasticMQ + Neon, which this
> setup intentionally replaces with the Docker dev-services container + goaws.
> Neon is **not** used; worktree #1 runs against the local Docker Postgres.

## Running the app (`bun dw`)

`bun dw` wraps `infisical run`, which is **not** available here, so call the script
directly (this is what `cloud-dev.sh` does):

```sh
bun scripts/dw/index.ts
```

For worktree #1 this runs the full stack inline via `scripts/dev.ts`:
server `:8080`, vite dashboard `:3000`, checkout `:3001`, workers, leaf `:3099`,
plus `trigger`. Multi-worktree `bun dw setup` (#N>1) needs Neon + Docker/portless
and is **not** wired up here — ignore it.

Non-blocking noise you can ignore: `trigger` spams "Waiting for you to login"
(trigger.dev cloud; background jobs only); `leaf` (AI chat/MCP) crashes with a
`ZodError` on missing env (`SLACK_*`, `FIRECRAWL_API_KEY`, its own `DATABASE_URL`);
`InvalidAccessKeyId`-style noise comes from the dummy `AWS_*=x` creds. None of
these block the dashboard, server API, workers, or checkout.

## Backing services & env (the gotchas)

- Backing services are **Docker containers + a goaws process**, not host services.
  `dockerd` has no systemd here — `cloud-dev.sh` starts it via `sudo nohup dockerd`.
  Docker uses the `fuse-overlayfs` storage driver + `iptables-legacy` (baked into
  the VM image). Use `sudo docker ...` if your shell isn't in the `docker` group.
- **ClickHouse is intentionally not run.** The server only *warns* about a missing
  ClickHouse/Tinybird URL (analytics only) — the dashboard + product CRUD work
  without it. Only `DATABASE_URL`, `ENCRYPTION_IV`, `ENCRYPTION_PASSWORD`,
  `BETTER_AUTH_SECRET`, `BETTER_AUTH_URL` are hard-required (see
  `server/src/utils/initUtils.ts`).
- `server/.env` is the source of truth for the server process (it runs from
  `server/`, so `loadLocalEnv` resolves `server/.env`). Any key NOT in the file
  falls through from the process environment, so Cursor Cloud secrets are picked
  up automatically. Optional keys for fuller functionality:
  `STRIPE_SANDBOX_SECRET_KEY`/`_WEBHOOK_SECRET`/`_CLIENT_ID` (checkout/attach),
  `ANTHROPIC_API_KEY` (AI feature naming), `RESEND_API_KEY`/`RESEND_DOMAIN`
  (email OTP delivery), `SVIX_API_KEY` (webhooks), `GOOGLE_CLIENT_*` (Google login).

## Migrations

The DB lives in a Docker volume (`autumn-dev-services_autumn-dev-postgres-18`) and
persists across reboots. After pulling code with new migrations, run from the repo
root with `DATABASE_URL` passed explicitly (the `bun db` npm alias wraps Infisical,
and `loadLocalEnv` resolves to the repo root — not `server/` — from `/workspace`):

```sh
AUTUMN_DB_DIRECT=1 DATABASE_URL=postgresql://postgres:postgres@localhost:5432/autumn bun db migrate
```

Add `--bootstrap` only on a fresh/empty DB (skips the index-DDL safety check). A
fresh Postgres also needs `CREATE EXTENSION pg_trgm` (done by `cloud-dev.sh`).
Balance SQL functions live in `server/src/internal/balances/utils/sql/*.sql`
(loaded via `psql` by `cloud-dev.sh`).

## Sign-in / OTP

Email OTP login. With no `RESEND_API_KEY`, the code is printed to the server log:

```sh
grep -a "SIGN IN OTP" <server output>
```

## Lint / typecheck / test

- **Typecheck:** `bun ts` (turbo) or scoped `bunx turbo run ts --filter=@autumn/server`.
- **Lint/format:** biome. `bunx biome check .` from the repo root currently **errors**
  (`packages/atmn/biome.json` is a nested *root* config) — this is a pre-existing
  repo quirk, not an env issue. Lint a scoped set instead, e.g.
  `bunx biome check server shared vite` (expect many pre-existing warnings).
- **Dead-code:** `bun knip` (husky pre-commit hook also runs this + scoped typechecks).
- **Tests:** `bun tw` is the **cloud test swarm** orchestrator — it needs a cloud
  provider (Modal/Vercel) + a Stripe key pool and is **not** runnable locally
  without those secrets. For local integration tests use
  `bun test:integration` (needs the services above + `server/.env`).
