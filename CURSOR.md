# CURSOR.md — Cursor Cloud agent instructions

Environment/setup notes for Cursor agents running this repo in a Cursor Cloud VM.
General repo rules live in `AGENTS.md` (generated by ai-sync from `ai/rules/`); this
file is specific to the Cursor Cloud environment.

## No Infisical — env comes from `server/.env`

This environment runs **without Infisical**. The `package.json` scripts that start
with `infisical run --env=dev -- …` (including `bun dev`, `bun dw`, `bun db`,
`bun t`) assume Infisical injects secrets. Here, env vars come from `server/.env`
instead, loaded over `process.env` by the bun preload `scripts/preload-env.ts` →
`loadLocalEnv`. Invoke the underlying scripts directly rather than the
Infisical-wrapped npm aliases.

## Backing services (local, installed in the VM snapshot)

Installed once via `scripts/setup/agent-bootstrap.sh` (Postgres 18, Redis Stack —
RedisJSON required, ClickHouse, JRE, ElasticMQ jar). They are NOT running on a fresh
boot — start them each session with:

```sh
bash scripts/setup/agent-services.sh
```

This starts Postgres (`:5432`, db `autumn`, `postgres`/`postgres`), Redis Stack
(`:6379`), ClickHouse (`:8123`), ElasticMQ (`:9324`) and (re)writes `server/.env`.
It ends with `error: Script not found "db:migrate"` — that line in the script is
stale (the real command is `bun db migrate`, with a space). Harmless: services and
`server/.env` are already set up by then and the DB is already migrated in the snapshot.

## Running the app (the `bun dw` entrypoint)

Use the dw script directly — NOT `bun dw` (the npm alias wraps `infisical run`):

```sh
bun scripts/dw/index.ts
```

In this canonical checkout (worktree #1) this runs the full dev stack inline via
`scripts/dev.ts`: server `:8080`, vite `:3000`, checkout `:3001`, workers, leaf
`:3099`, plus `trigger` and (if installed) the Stripe CLI. `bun dev` is equivalent
for worktree #1. Multi-worktree `bun dw setup` (#N>1) needs Neon (`NEON_API_KEY`) +
Docker/portless and is NOT wired up here.

Non-blocking noise you can ignore: `trigger` spams "Waiting for you to login"
(trigger.dev cloud; background jobs only); `leaf` (AI chat/MCP) crashes on missing
env (`SLACK_*`, `FIRECRAWL_API_KEY`, its own `DATABASE_URL`); `InvalidAccessKeyId`
warnings come from the dummy `AWS_ACCESS_KEY_ID=x` (S3 edge config unavailable locally).

## Migrations

The DB persists in the snapshot, so you usually don't need to migrate. After pulling
code with new migrations (bypass the Infisical wrap; DATABASE_URL must be passed
explicitly because `loadLocalEnv` resolves to the repo root, not `server/`, from `/workspace`):

```sh
AUTUMN_DB_DIRECT=1 DATABASE_URL=postgresql://postgres:postgres@localhost:5432/autumn bun db migrate
```

Add `--bootstrap` only on a fresh/empty DB. Balance SQL functions live in
`server/src/internal/balances/utils/sql/*.sql` (loaded via `psql` on first provision).

## Sign-in / OTP

Email OTP login. With no `RESEND_API_KEY`, the code is printed to the server log —
`grep -a "SIGN IN OTP" <server output>`.

## Secrets (env vars)

`server/.env` is the source of truth (local-service URLs + auto-generated
`BETTER_AUTH_SECRET` / `ENCRYPTION_*`). The empty external-integration lines were
removed from `server/.env` on purpose: any key NOT present in the file falls through
from the process environment, so Cursor Cloud secrets are picked up automatically.
The dashboard + product creation work with NO external secrets. Optional keys for
fuller functionality: `STRIPE_SANDBOX_SECRET_KEY`, `STRIPE_SANDBOX_WEBHOOK_SECRET`,
`STRIPE_SANDBOX_CLIENT_ID` (checkout/attach), `ANTHROPIC_API_KEY` (feature names),
`RESEND_API_KEY`/`RESEND_DOMAIN` (email OTP), `SVIX_API_KEY` (webhooks),
`GOOGLE_CLIENT_ID`/`GOOGLE_CLIENT_SECRET` (Google login).

## Lint / typecheck / test

- Lint/format: `bunx biome check .` (config `biome.json`).
- Dead-code: `bun knip` (run by the husky pre-commit hook).
- Typecheck: `bun ts` (or `bunx turbo run ts --filter=@autumn/server`).
- Integration tests: `bun test:integration` (needs services + `server/.env`).

## GUI testing

`localhost` is reachable in Chrome only when launched with `--no-sandbox` in this VM.
